2. AUTHENTICATION FLOW
================================================================

- Uses Supabase Auth (email/password)
- On signup, a Postgres trigger (handle_new_user) auto-creates a 
  profile row with role and full_name from user metadata
- AuthContext stores the authenticated user globally via React Context
- On login: credentials → Supabase Auth → fetch profile → role-based redirect
- Token is managed by Supabase JS client (auto-refreshes, stored in localStorage)
- Token is automatically attached to all Supabase requests via the client
- Unauthenticated users are redirected to /auth by DashboardLayout
- On logout: clear user state → redirect to /auth


3. ROLE-BASED ROUTING
================================================================

Teacher routes:
  /teacher/dashboard    → TeacherDashboard
  /teacher/upload       → UploadForm
  /teacher/content      → TeacherContent

Principal routes:
  /principal/dashboard  → PrincipalDashboard
  /principal/pending    → PendingApprovals
  /principal/content    → AllContent

Public routes:
  /                     → RootRedirect (auth-aware: redirects to dashboard or landing)
  /landing              → LandingPage
  /auth                 → AuthForm (login/signup)
  /live/:teacherId      → LiveBroadcast (no auth required)

Protected routes are enforced by DashboardLayout:
  - Checks if user is authenticated
  - Checks if user's role matches the allowedRole prop
  - Redirects to the correct dashboard if role mismatch
  - Shows skeleton loading state while checking auth


4. API INTEGRATION APPROACH
================================================================

All API calls go through a dedicated service layer:
  - services/auth.service.js     → login, signup, logout, getCurrentUser, updateProfile
  - services/content.service.js  → uploadContent, getMyContent, getAllContent, 
                                    getPendingContent, updateContentStatus, deleteContent,
                                    getLiveContent, getStats
  - services/approval.service.js → getPendingContent, approveContent, rejectContent,
                                    getApprovalStats

NO direct Supabase calls are made inside components.

Error handling pattern:
  - Services throw errors on failure
  - Components catch errors and display toast notifications
  - Loading states are tracked per-component via useState

The service layer is easily replaceable — swap Supabase calls with 
any REST/GraphQL client without touching components.


5. STATE MANAGEMENT APPROACH
================================================================

- Global state: React Context (AuthContext) for user/auth only
- Local state: useState for component-specific data (content lists, filters, etc.)
- No external state library (Redux, Zustand) needed — app complexity doesn't warrant it
- Form state: React Hook Form with Zod validation
- Server state: Fetched on mount via useEffect, refetched after mutations

Design principles:
  - Keep state minimal and close to where it's used
  - Avoid prop drilling via Context for cross-cutting concerns
  - Derive computed values (filtered lists, stats) instead of storing duplicates


6. ASSUMPTIONS
================================================================

- Supabase is the backend — auth, database, and file storage
- Row Level Security (RLS) enforces authorization server-side
- File uploads are limited to images (JPG, PNG, GIF) with 10MB max
- Content rotation on the live page uses client-side interval timers
- The live broadcast page polls for new content every 60 seconds
- Teachers can only see/manage their own content
- Principals can see all content and approve/reject any submission
- Rejection requires a mandatory reason (enforced in UI and modal)
- The app is a client-side SPA (Vite + React), not SSR
- All authorization is enforced server-side via Supabase RLS policies
