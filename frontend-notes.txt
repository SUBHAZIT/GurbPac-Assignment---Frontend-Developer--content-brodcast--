2. AUTHENTICATION FLOW
================================================================

- Supabase Auth (email/password)
- On signup, a Postgres trigger auto-creates a profile row
- Public signup only creates 'viewer' accounts
- Teacher/principal accounts created only by principals via Manage Staff
- AuthContext stores user globally via React Context
- Login → Supabase Auth → fetch profile → role-based redirect:
    principal → /principal/dashboard
    teacher  → /teacher/dashboard
    viewer   → /live/all
- Token auto-managed by Supabase JS client (localStorage + auto-refresh)
- DashboardLayout guards all protected routes
- Logout → clear state → redirect to /auth


3. ROLE-BASED ROUTING
================================================================

Teacher routes:
  /teacher/dashboard    → TeacherDashboard
  /teacher/upload       → UploadForm
  /teacher/content      → TeacherContent

Principal routes:
  /principal/dashboard  → PrincipalDashboard
  /principal/pending    → PendingApprovals
  /principal/content    → AllContent
  /principal/manage     → ManageTeachers (create staff)

Shared routes:
  /profile              → Profile edit (all roles)

Public routes:
  /                     → RootRedirect
  /landing              → LandingPage
  /auth                 → AuthForm (login + viewer signup)
  /live/:teacherId      → LiveBroadcast (10 free views, then login)

Protected routes enforced by DashboardLayout:
  - Checks authenticated
  - Checks role matches allowedRole
  - Redirects on mismatch
  - Shows skeleton during check


4. API INTEGRATION APPROACH
================================================================

All API calls go through a dedicated service layer:
  - auth.service.js     → login, signup, logout, getCurrentUser,
                           getSession, updateProfile
  - content.service.js  → uploadContent, getMyContent, getAllContent,
                           getPendingContent, updateContentStatus,
                           deleteContent, getLiveContent, getStats
  - approval.service.js → getPendingContent, approveContent,
                           rejectContent, getApprovalStats
  - viewer.service.js   → recordView, getViewCount, checkViewAccess

NO direct Supabase calls inside components.

Error handling:
  - Services throw errors on failure
  - Components catch + show toast notifications
  - Loading states tracked per-component via useState

API layer is replaceable — swap Supabase with any REST/GraphQL
backend without touching components.


5. STATE MANAGEMENT APPROACH
================================================================

- Global state: React Context (AuthContext) for user/auth only
  Exposes: user, loading, login, logout, refreshUser
- Local state: useState for component-specific data
- Form state: React Hook Form (independent from component state)
- No Redux/Zustand — complexity doesn't warrant it

Principles:
  - State kept close to where it's used
  - No prop drilling via Context
  - Derived values computed, not stored
  - refreshUser() for post-edit profile updates


6. VIEW GATING (PUBLIC BROADCAST)
================================================================

Anonymous viewers get 10 free unique content views:
  - Tracked server-side in public_views table (Supabase)
  - Browser fingerprint (navigator + screen + timezone hash) as viewer ID
  - RLS allows INSERT/SELECT but NOT UPDATE/DELETE (tamper-proof)
  - Server count is authoritative — clearing cookies doesn't reset
  - After 10 views → paywall screen → sign up as viewer
  - Logged-in users get unlimited access


7. STAFF MANAGEMENT
================================================================

- Public signup → viewer only (no role selector)
- Teacher/principal accounts → created by principals only
- Manage Staff page: /principal/manage
  - Form creates user via Supabase Auth with role metadata
  - Trigger auto-creates profile row
  - Success modal shows credentials with copy-to-clipboard
  - Credentials shown only once
  - Staff list with role badges and join dates


8. ASSUMPTIONS
================================================================

- Supabase is the backend (auth, database, storage)
- Row Level Security (RLS) enforces all authorization server-side
- File uploads: JPG, PNG, GIF only, 10MB max
- Content rotation: client-side interval timers
- Live broadcast polls every 60 seconds
- Teachers see/manage own content only
- Principals see all content, approve/reject any
- Viewers watch broadcasts only
- Rejection requires mandatory reason
- App is a client-side SPA (Vite + React), not SSR
- All auth enforced server-side via RLS
